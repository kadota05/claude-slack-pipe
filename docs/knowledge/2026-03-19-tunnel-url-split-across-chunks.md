# ストリーミングチャンクでURLが分割される問題

## 症状

Slackにlocalhost URLがそのまま表示され、トンネルURLへの書き換えが行われない。トンネルは確立されるが、タイムアウト後に確立されるため、書き換えに間に合わない。

## 根本原因

2つの問題が同時に発生していた：

1. **handleTextでURLを検知できない**: ストリーミングテキストは小さなチャンクで届く。URLが `http://local` + `host:52227` のように分割されると、新しいチャンクのみをスキャンする最適化（`extractLocalUrls(text)`）ではURLを検知できない。結果、トンネルの先行開始が行われない。

2. **タイムアウトが短すぎる**: handleResultでの8秒タイムアウト内にトンネルが確立されなかった。実測で9秒以上かかるケースがあり、8秒では不十分。

## 証拠

Bridgeログ：
```
02:25:53 — result処理開始、localUrls検知、トンネル開始（handleTextでは先行開始されていない）
02:26:01 — 8秒タイムアウト（urlMap空のまま）
02:26:02 — トンネル確立（1秒遅い）
```

## 修正内容

1. `handleText`を`extractLocalUrls(this.textBuffer)`に戻し、全バッファをスキャン（TunnelManagerの重複排除により性能影響は無視可能）
2. `TUNNEL_TIMEOUT_MS`を8000→15000に増加

## 教訓

- ストリーミングテキストでの文字列検索は、チャンク境界での分割を常に考慮する。「新しいチャンクだけスキャン」は一見効率的だが、パターンが分割されるリスクがある。
- タイムアウト値は実測値に十分なマージンを持たせる。初回計測で5秒でも、ネットワーク状況によって倍近くかかることがある。
