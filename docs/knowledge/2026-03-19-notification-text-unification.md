# 通知テキスト統一・装飾整理

## 症状

Slackユーザーが体験していた問題：

1. **二重通知**: 応答完了時に「Complete」というフッターpostMessageの通知と、最終テキスト回答の通知が同時に来る。どちらも「回答終わった」という同じ意味なのに通知が2回鳴る
2. **意味不明な通知文**: `bundle collapsed` というupdate fallback textが意味不明。思考中であることが伝わらない
3. **テキスト応答の残留フッター**: `応答中...` のcontextブロックがテキスト応答に付くが、bundleのフッター（`実行中...`、`思考中...`）と違ってcollapseで消えないため永続的に残り続ける
4. **絵文字の重複**: `:white_check_mark:` がリアクション（完了通知）と装飾（ツール完了アイコン）の両方で使われており、意味が混同する
5. **英語の通知**: `Complete` や `bundle collapsed` が英語のまま

## 根本原因

- **通知テキスト戦略の欠如**: 各所（group-tracker, stream-processor, index.ts）でtext値をその場でハードコードしていたため、一貫性がなかった
- **postMessageのtext field = Slack通知のプレビュー文**: この認識が設計時に欠けていた。`text: 'Complete'` は単なるfallbackではなく、ユーザーのスマホに「Complete」と表示される
- **リアクション vs 装飾の設計分離なし**: 絵文字を場当たり的に選んでいたため重複が発生

## 修正内容

### アプローチ: `notification-text.ts` モジュールの導入

テキスト生成と絵文字定義を1ファイルに集約し、全箇所がそのモジュールを参照するよう変更。

### 絵文字の完全分離

| 用途 | 絵文字 | ルール |
|------|--------|--------|
| リアクション専用 | ⏳🧠✅ | メッセージ本文内では絶対に使わない |
| 装飾専用 | 💭🔧🤖✓✗ | リアクションでは絶対に使わない |

### 通知テキストの統一

| 場面 | 変更前 | 変更後 |
|------|--------|--------|
| フッターpostMessage | `Complete` | `opus \| 1,234 tokens \| 3.2s` |
| bundle collapse | `bundle collapsed` | `💭×1  🔧×3 (2.5s)` |
| bundle初期化 | `''`（空文字） | `...` |
| ツール更新 | `3ツール実行中` | `🔧 Read, Bash, Grep` |
| 思考更新 | `思考中...` | `💭 思考中` |
| テキスト応答 | `textBuffer.slice(0, 100)` | `notifyText.text(buffer)` |

### フッター削除

- テキスト応答の「`:hourglass_flowing_sand: 応答中...`」contextブロックを削除
- ツール/思考のフッターは**残した**（理由: bundleに折りたたまれると消えるが、テキストフッターは永続的に残り続けるため）

## 教訓

1. **Slack APIのtext fieldは通知プレビュー文**: postMessageでもupdateでも、`text` fieldはSlackの通知バナーやメッセージ一覧のプレビューに表示される。ハードコードせず、常にユーザーにとって意味のある内容にすべき
2. **絵文字の用途を明示的にカテゴリ分けする**: リアクション（Slackのリアクション機能）と装飾（メッセージ本文内のアイコン）で同じ絵文字を使うと、ユーザーにとって意味が混同する。カテゴリごとに排他的に割り当てる
3. **テキスト生成は散在させない**: 通知文言が5箇所以上にハードコードされると一貫性を保てない。1モジュールに集約すれば変更も1箇所で済む
4. **中間装飾は「消えるかどうか」で判断する**: bundleに折りたたまれて消えるフッターはUX上問題にならないが、永続するフッターは「処理完了後も残る不要な情報」になる
