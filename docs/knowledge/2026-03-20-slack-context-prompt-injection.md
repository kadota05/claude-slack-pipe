# Slack環境コンテキストのプロンプト注入

## 症状

Claude CLI が生成する図（テーブル、ASCII ボックス図、ツリー図など）が Slack mobile の狭い画面で崩れる。また、Claude は自分が Slack 経由で応答していることを知らないため、PC前提の出力（ログ確認を促す、システムプロンプト承認を求める等）をしてしまう。

## 根本原因

Claude CLI はターミナル直接利用を前提としており、Bridge 経由で Slack にメッセージが転送されている文脈を持っていなかった。

## 修正内容

### 構成

- `src/bridge/slack-context.ts` — コンテキストプレフィックス定数 `SLACK_CONTEXT_PREFIX` を定義
- `src/bridge/persistent-session.ts` — `sendInitialPrompt()` でユーザーメッセージの前に prepend

### 注入タイミング

- **初回のみ**（`sendInitialPrompt()`）。毎ターン入れる方式も検討したが、初回で十分と判断。
- `sendPrompt()`（通常ターン）では素のユーザーメッセージをそのまま送る。

### プロンプト文面のポイント

- 「普段は CLI から使われているが、今は Slack 経由」という対比で状況を伝える
- 箇条書きではなく一続きの文章にして、情報の流れを自然にする
- 制約だけ列挙しても Claude は文脈を理解できない。「なぜそうなのか」を含めることで推論の質が上がる
- `[System Context]` のようなラベルは避け、`[Slack Bridge Context]` のようにユーザーメッセージの補足であることが明確な名前にする

### 幅の制限値

- **45文字**を採用。40文字（保守的すぎて情報量が減る）と50文字（小型スマホで崩れる）の間を取った。
- 実際にテーブル・ボックス図・ツリー図を40/50文字で生成して比較した上で決定。

## 教訓

1. **AI へのプロンプトは「制約リスト」ではなく「状況説明」にする** — 箇条書きで制約を並べるより、「何が起きているか」を narrative として伝える方が Claude の理解度が上がる
2. **プロンプト文面は実際に生成結果を見て調整する** — 幅制限は理論値ではなく、実際の出力を比較して決めた
3. **注入箇所はメソッド内部で行う** — 呼び出し元（session-coordinator.ts 等）を変更不要にすることで影響範囲を最小化
4. **トークン消費は100以内に収める** — 毎ターン or 初回のみに関わらず、プレフィックスは軽量に保つ
