# README書き直し設計

## 目的

READMEを「読みやすく、流れるように頭に入る」ドキュメントに書き直す。

## 問題点

- plaintext図が多すぎて文章の流れが分断される
- テキスト同士の繋がりが弱く、羅列的
- 冒頭で仕組みが伝わらない

## 方針

### 構成（順序）

1. **冒頭（概要）** — 1文で何か + 1文で仕組み + コンパクトなデモ図
2. **使い方** — 文章ベース。図は最小限でテキストの流れに溶け込ませる
3. **セットアップ** — 自動/手動をコンパクトに
4. **アーキテクチャ** — 開発者向け。ここだけ構造図を使う
5. **トラブルシューティング** — テーブル形式（現状維持）
6. **開発** — コマンド一覧（現状維持）

### トーン

プロダクト紹介ページ的。簡潔で読み手を引き込む。箇条書きの羅列ではなく段落で語る。

### 図のルール

- **最大3つ** — 冒頭デモ、アーキテクチャ概念図、（必要なら1つ）
- **箱線図は使わない** — `┌─────┐` のような大きな箱は廃止。インデントベースのシンプルな表現にする
- **図の前後に文脈を持たせる** — 図が文章の流れの中に自然に溶け込むこと

### 冒頭セクション

```markdown
# Claude Code Slack Bridge

SlackのDMからClaude Codeを操作するブリッジ。Node.jsプロセスが
Slack BoltとClaude CLIの子プロセスをstdin/stdoutで接続し、
ターミナルで使うClaude Codeの体験をそのままSlack上で再現します。

DMを送るだけでセッションが始まり、ツール実行や思考過程が
リアルタイムにスレッドへストリーミングされます。

  You: このバグを調査して
    💭×2 🔧×3 (1.2s)
    💬 「調査結果です。原因は...」
    💭×1 🔧×5 (3.4s)
    💬 「修正しました」
    📊 Tokens: 12,345 | $0.15 | 45s

  You: テストも書いて    ← 同じスレッドで文脈を保持
```

### 使い方セクション

段落ベースで書く。Home Tab → DM → リアクション → コマンドの流れを文章で繋ぐ。

- Home Tabの説明は図なしで「モデルとディレクトリを選ぶ」と1段落で伝える
- DMの流れは「DMを送る → スレッドに返信 → 1スレッド=1セッション」と繋げる
- リアクションは `⏳→🧠→✅` のインライン表現 + 🔴中断の説明を自然に繋げる
- コマンドはテーブル（現状のまま、コンパクト）

### アーキテクチャセクション

冒頭で概念を1文で伝えた上で、シンプルな構造図を1つだけ置く。

```
Slack ←WebSocket→ Node.js (Bolt) ←stdin/stdout→ claude -p (子プロセス)
```

この1行が本質。その下に補足としてモジュール構成テーブルとデータフロー（文章ベース）を置く。

### セットアップセクション

現状の内容を維持。自動セットアップ（推奨）を先頭に置く構成は変更なし。

### 削除・縮小する要素

- 冒頭の大きな箱線図 → インデントベースのコンパクトなデモに
- Home Tab箱線図 → 文章に置き換え
- DM/Thread箱線図 → 文章に置き換え
- ツール実行詳細のモーダル箱線図 → 文章で簡潔に説明
- アーキテクチャの大きな箱線図 → 1行の概念図 + モジュールテーブル
- データフローの番号付きリスト → 段落ベースの説明
- セッションライフサイクルの状態遷移図 → 文章に溶かす
